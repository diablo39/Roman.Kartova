### Overview

The slice wires 10 user-initiated Organization HTTP mutations to `IAuditWriter.AppendAsync(...)` on their committed-success path, adds `ICurrentUser.DisplayName` (JWT-claim snapshot feeding `audit_log.actor_display`), and introduces the `OrganizationAuditActions` / `AuditTargetTypes` taxonomy. The implementation matches the spec closely: every `AppendAsync` call sits **below** all early-return guards (verified per handler), so rejected mutations write no row; old-value snapshots are captured before the mutating call (`ChangeMemberRole`, `UpdateTeamMember`) or before hard-delete (`Offboard`, `DeleteTeam`); all `data` values are strings (jsonb hash-stability honored); and the cross-module `Audit.Infrastructure` ProjectReference is confined to the test project (ADR-0082 intact — no production Organization project gained it). The in-transaction fail-closed semantics from Phase 1 (ADR-0018 / ADR-0090) are reused unchanged: `AppendAsync` rides the shared `ITenantScope` connection via `AddModuleDbContext`, so an audit-write failure rolls the business change back. Gate-5 evidence is three real-seam integration tests (happy role-change with chain assertion, offboard-snapshot-survives-delete, last-OrgAdmin negative writes-no-row) plus a 4-case unit test for the claim fallback. I found no blocking-class defects; a small number of should-fix and missing-test items are below.

### Should-fix issues

1. **`CreateInvitationHandler` appends the audit row in a separate transaction from the business write it claims to audit atomically.**
   `src/Modules/Organization/Kartova.Organization.Infrastructure/CreateInvitationHandler.cs:125` does `await db.SaveChangesAsync(ct)` inside its own `try/catch (DbUpdateException)`, and the `await audit.AppendAsync(...)` is at line 182 — *after* the catch blocks have returned. That is the correct only-on-success placement (good), and because both the business save and the audit write share the same per-request `ITenantScope` transaction (ADR-0090), they still commit/roll-back together at request end. **However**, unlike the other 9 handlers, the invitation business state is already flushed to the DB connection (line 125) *before* the audit append; if `AppendAsync` throws, the surrounding request transaction rolls back the flushed invitation+user rows — correct — but the handler's own compensation logic (best-effort KC `DeleteUserAsync`) does **not** run for an audit-write failure, so a KC user would be orphaned with no DB invitation and no compensation. Spec §3 decision 1 promises "fail-closed … the business change rolls back"; the DB side does roll back, but the KC side (which the handler goes to great length to compensate elsewhere) is left orphaned on the audit-failure path specifically. *Impact:* an `audit_log` outage during invitation create orphans a KeyCloak user. *Fix:* wrap the `AppendAsync` for this handler so an audit failure triggers the same best-effort `kc.DeleteUserAsync(kcId, ct)` compensation before rethrowing — or document in the handler XML that audit-failure KC orphaning is an accepted consequence of fail-closed (consistent with foundation §6's "audit outage blocks audited mutations"). *Test that would catch it:* an integration test that forces `AppendAsync` to throw on the invitation path and asserts no orphaned KC user remains (or asserts the documented accepted-orphan behavior).

2. **`AuditWriter` XML doc still claims "Phase 1 writes `User` only (wired callers arrive in Phase 2)".**
   `src/Modules/Audit/Kartova.Audit.Infrastructure/AuditWriter.cs:19-21` — this slice *is* Phase 2 and wires the callers, yet the class summary still reads "wired callers arrive in Phase 2." It is now stale/misleading: the writer still hard-codes `AuditActorType.User` (line 60) and `currentUser.DisplayName` will **throw** if ever invoked with no HTTP principal, which is the documented gap the spec defers (System-actor sweep). *Impact:* a future maintainer reading this comment may believe System-actor support already exists or that no callers are wired yet. *Fix:* update the comment to "All wired callers (Phase 2) are authenticated HTTP requests writing `User`; the `System`-actor path is deferred (see audit-event-wiring spec §2). `DisplayName` throws if resolved with no HTTP principal — by design, since no non-HTTP caller is wired." *Test:* n/a (doc-only).

3. **No regression coverage that the 7 non-member, non-asserting handlers still emit a row (or that their existing happy-path tests survived the new audit write).**
   Tasks 4–5 wired `CreateTeam/UpdateTeam/DeleteTeam/AddTeamMember/RemoveTeamMember/UpdateTeamMember/UpdateOrgProfile` and added `subject:` overrides to their existing integration tests (e.g. `CreateTeamTests.cs:892`, `AddTeamMemberTests.cs:659`) — but those existing tests assert only the HTTP response, not that the audit row landed. Because the audit write is now *in the same transaction*, a broken audit write (e.g. a bad `targetType` constant, or a future RLS regression) on one of these 7 handlers would surface as a 500 in those tests — so they are not entirely blind. The spec §7 explicitly **defers** per-handler payload assertions for these 8 handlers, so this is consistent with the agreed scope, not a violation. *Impact:* a payload-shape bug (wrong `data` key) in any of the 7 deferred handlers ships unguarded until the follow-up. *Fix (or accept):* either add one parameterized integration test asserting `action`/`targetType`/`targetId` for each of the 7 (cheap — `ReadAuditLogAsync` already exists), or confirm the deferral is intentional for this slice. Flagged as should-fix because the cost is low and the taxonomy constants are the stable on-disk contract.

### Missing tests

1. **Fail-closed on a wired Phase-2 handler.** Phase 1 has the mandatory rollback test for the *mechanism*, but no Phase-2 test proves that when `AppendAsync` throws, a wired Organization mutation (e.g. role change) rolls its business change back and returns 500. The spec leans on Phase 1's proof, which is reasonable, but a single end-to-end fail-closed test on one wired handler would close the "does the wiring actually preserve fail-closed?" question (and would catch the `CreateInvitation` KC-orphan gap in Should-fix #1). *Add:* force the writer to throw (e.g. seed a duplicate `(tenant_id, seq)` or inject a throwing `IAuditWriter` decorator in the test host) and assert the `users.RealmRole` / invitation row is unchanged after the failed request.
2. **`actor_display` fallback chain at the HTTP seam.** The unit test (`HttpContextCurrentUserDisplayNameTests.cs`) covers all 4 claim-fallback branches in isolation, and the happy integration tests assert `actor_display == name`. No integration test exercises the `preferred_username`/`email`/`sub` fallback through the real JWT pipeline — the signer only sets `name`. Low priority (unit-covered), but a single integration assertion with a `name`-less token would prove the real `JwtBearer` claim mapping preserves `preferred_username`.

### What looks good (3–5 specific items)

1. **Offboard pre-delete snapshot is correct and provably so.** `OffboardMemberHandler.cs:48-49` captures `target.DisplayName`/`target.Email` *before* `db.Users.Remove(target)` (line 64) and appends *after* `SaveChangesAsync` (line 67), exactly per spec §3 decision 5 / ADR-0102. The dedicated integration test `Offboard_WritesSnapshotThatSurvivesTargetDeletion` (`AuditWiringTests.cs:732`) asserts the snapshot fields are non-empty after the same-transaction hard-delete — the right test for the right risk.
2. **`org.profile_updated` target id is the tenant id, matching spec §4.** `UpdateOrgProfileHandler.cs:55` uses `org.Id.Value`; `Organization.Create` sets `tenantId = new TenantId(id.Value)` (`Organization.cs:35`), so `org.Id.Value` *is* the tenant id — the spec's "tenantId" and the code agree by construction. Verified rather than assumed.
3. **Success-path-only placement is consistent across all 10 handlers.** Every `AppendAsync` sits below every early-return guard — `ChangeMemberRoleHandler` (after the `LastOrgAdmin`/`NotFound`/`InvalidRole` returns), `DeleteTeamHandler` (only the `Deleted` branch, after the `appCount > 0` and `NotFound` returns), `AddTeamMemberHandler.cs:41` (after `TeamNotFound`/`AlreadyMember`), `RemoveTeamMemberHandler.cs:31`, `CreateInvitationHandler.cs:182` (after all `Failed(...)` returns). The negative test `ChangeRole_LastOrgAdminGuard_WritesNoAuditRow` proves the principle for the guarded case.
4. **Chain-integrity assertion in the happy path.** `AuditWiringTests.cs:783` `AssertChainLinked` re-walks every row, asserting contiguous `seq` from 1 and `prev_hash == predecessor.row_hash` (genesis = 32 zero bytes) — this verifies the slice did not break the Phase-1 hash chain (ADR-0018) at the new write sites, not merely that a row exists.
5. **ADR-0082 boundary respected with an explained test-only exception.** The `Audit.Infrastructure` ProjectReference is added only to `Kartova.Organization.IntegrationTests.csproj:38` with an inline justification (needs `AuditDbContext` migration in the shared container); no production Organization project references it. Handlers depend solely on the SharedKernel `IAuditWriter`. The arch suite (NetArchTest) targets `src/` assemblies, so this stays green.
